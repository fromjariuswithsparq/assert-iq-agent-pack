The shift from QA to QI

Traditional QA asks one question: did this pass? Quality Intelligence asks four — and synthesizes them into Decision Confidence no single coverage percentage or green CI badge can give you.

LAYER 01

Change Risk

What moved, and how broadly does it reach?

LAYER 02

Protection Strength

Do the right tests actually cover it?

LAYER 03

Signal Trustworthiness

Can we believe the results we're seeing?

LAYER 04

Outcome Evidence

What do escaped defects and telemetry say?

The immediate impact

  • Engineers stop asking "do we have tests?" and start asking "do our tests protect the right things?"
  • Release conversations shift from gut-feel to evidence.
  • Escaped defects get root-caused at the signal layer they slipped through, not just triaged.
  • QA stops being a gate at the end of a sprint and becomes a continuous signal woven into every PR, every merge, every retrospective.

Get started in three steps

Install the pack

Two install paths. Pick the one that matches how comfortable you are dropping the pack into your team's codebase.

Path A · Try it on the pack

Best if you want to play with Assert.IQ before touching your team's repository. Clone the pack, run the installer, and open the pack folder as your workspace.

git clone https://github.com/fromjariuswithsparq/assert-iq-agent-pack
cd assert-iq-agent-pack
bash install.sh           # macOS / Linux / WSL
# or
pwsh ./install.ps1        # Windows PS 7+

Reverse with bash install.sh --uninstall.

Path B · Install into your codebase

The real deployment path. Run the bootstrap script from a terminal in your target repo — no editor required:

# 1. Clone the pack somewhere (one time, anywhere)
git clone https://github.com/fromjariuswithsparq/assert-iq-agent-pack ~/assert-iq-agent-pack

# 2. cd into YOUR repo
cd ~/code/my-app

# 3. Run the bootstrap script from the clone
bash ~/assert-iq-agent-pack/scripts/bootstrap.sh --mode=trial
# Windows: pwsh -File ~\assert-iq-agent-pack\scripts\bootstrap.ps1 -Mode trial

The script is fully standalone — it accepts --preset=solo|pod, prompts interactively, and writes everything Copilot and Claude need. No chicken-and-egg. You do not need to open VS Code or Claude Code first, and the /assert-iq-bootstrap skill does not need to be loaded — the script is what the skill calls under the hood.

Where does --preset=solo put the QI instructions?

Solo is designed for a single developer who wants the QI rules to apply to every repo they open, not just this one. The instruction files (qi-foundation, qi-test-design, etc.) and CLAUDE.md install to your VS Code user prompts folder (~/Library/Application Support/Code/User/prompts/ on macOS, ~/.config/Code/User/prompts/ on Linux, %APPDATA%\Code\User\prompts\ on Windows) and ~/.claude/CLAUDE.md respectively — not to .github/instructions/ in the workspace.

Use --preset=pod if you want the instructions checked into this repo for the whole team.

Already have the pack loaded (cloned pack opened in your editor, or skills installed user-globally to ~/.agents/skills/)? You can also run /assert-iq-bootstrap from chat — same outcome.

--mode=trial is the safe default — files land in your workspace, but paths are added to .git/info/exclude. Your codebase's .gitignore is never touched.

Bootstrap writes twelve surfaces: .assert-iq/, .github/instructions/, .github/copilot-instructions.md, .github/skills/, .github/agents/, .claude/agents/, .claude/skills (symlink), .claude/settings.json, CLAUDE.md, AGENTS.md, .vscode/settings.json + .vscode/mcp.json, and hooks/. Pre-existing user files are snapshotted to <file>.assert-iq.pre-install for byte-for-byte restore on uninstall.

Don't want trial mode? Want skills in every workspace?

Use --preset=portable. Skills install user-globally to ~/.agents/skills/ (VS Code Copilot Chat) and ~/.claude/skills/ (Claude Code), so every repo you open has the 24 QI skills available. Workspace footprint is minimal — only the Assert-IQ chat agent files (.github/agents/, .claude/agents/) and the manifest land in the repo; no instructions, hooks, settings, or MCP config.

bash ~/assert-iq-agent-pack/scripts/bootstrap.sh --preset=portable

Reverse with bash ~/assert-iq-agent-pack/scripts/bootstrap.sh --uninstall --user.

Presets vs. Modes: What's the difference?
  • Presets (--preset) control Placement: Where do the files physically go on your hard drive? (Global OS directories vs. local workspace folders).
  • Modes (--mode) control Git Visibility: For the files that do land in your workspace, how does git treat them?

You mix and match them. For example, --preset=pod --mode=trial means "Put everything in my workspace (pod), but hide them in `.git/info/exclude` so I can evaluate them locally without bothering my team (trial)."

Once you're ready to share with the team, you use --mode=committed (or run --graduate), meaning "Keep the files in the workspace, but now let git track them so my team sees them."

Customize and wire everything in

Set your maturity tier

Open .assert-iq/maturity-profile.md and choose Early, Mid, or Higher. Agents read this on every quality and release question and scale their behavior accordingly.

Set your governance posture

Open .assert-iq/governance.md — compliance constraints the agents must respect (data handling, naming, refusal rules).

Wire your tools

The pack ships 20 pre-configured MCP servers in .vscode/mcp.json. Add credentials when VS Code prompts — they go to your OS keychain, not the file. See .vscode/MCP.html for a per-server setup guide.

Tailor config.yaml

Open .assert-iq/config.yaml. Edit manually — every field has inline comments — or attach the file to chat and say:

Customize this config.yaml file to my codebase and workspace.

The agent will ask a few targeted questions about your stack, tracker, and team, then fill in the placeholders.

Upgrade

Upgrades are explicit and intentional.

  1. Read the Releases page for what changed and any migration notes.
  2. Uninstall the current version where you installed it:
    • Path A (pack-as-workspace): bash install.sh --uninstall
    • Path B (codebase install): bash scripts/bootstrap.sh --uninstall in each target repo
  3. git pull (or re-clone) to the new tag, then re-run the same path to refresh.
